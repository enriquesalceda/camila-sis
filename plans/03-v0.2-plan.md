# Super Camila Sis — v0.2 Visual Overhaul Plan

## Context

v0.1 shipped a playable platformer using procedural rectangles for everything (ground, platforms, Camila's body, enemies, pickups, flag, HUD). The user just played it with his daughter and the placeholder graphics need a serious upgrade before her classroom demo. The art is the only thing changing — physics, level ASCII, controls, scene/entity architecture all stay. The session's goal is to get every visible pixel onto a "real game" footing: Kenney's CC0 Pixel Platformer pack for environment tiles + parallax backgrounds, hand-authored pixel art for the chef-themed entities Kenney doesn't cover (Camila's body, notebook, insect, nugget, chef-hat flag, scoop), proper sprite animations (walk cycles, bobbing, flag wave), a pixel font for HUD text, a Mario-style camera dead-zone instead of always-centered follow, and a level extended to 3-4 screens so the camera scrolling actually means something.

## Decisions made (override before approving if any are wrong)

1. **Kenney "Pixel Platformer" pack v1.2** — the spec mentions it by name; downloaded and inventoried. CC0. 180 main tiles (18×18), 24 background tiles (24×24), 27 character tiles (24×24). Direct zip URL in §4.
2. **Pick TILE = 18 (not 32)** — match Kenney's native size. The level ASCII glyph spacing stays the same; just the world unit shrinks. Kaplay renders pixel-art at 1:1 with `pixelDensity` and `image-rendering: pixelated`. Camila scales accordingly.
3. **Hand-authored sprites use a JS-array-to-canvas pixel painter**, not external PNGs. Each sprite's pixels are 2D string arrays with a palette dict. A helper renders them into an offscreen `<canvas>` and registers them as Kaplay sprites at load time. Reasoning: kid-readable (Camila can change the pixel grid in code and see results), no external editor, no extra deps, all art in version control. Lives in `src/pixel-art.js`.
4. **"Press Start 2P" font, bundled locally** as a TTF in `public/fonts/` and loaded via Kaplay's `loadFont()`. SIL Open Font License 1.1, bundle is allowed by OFL. ~30 KB. Used for HUD, menu, win, gameover text. Kenney's number tiles are NOT used for text — too narrow a use case to justify the dual-system.
5. **Don't replace Camila's face PNGs** — they're the user's actual photos of his daughter and the heart of the project. The body around them gets redone.
6. **Level expansion**: extend the existing ASCII string in `src/levels/level1.js` to ~160 chars wide (5× current). Parser is untouched — only the data string grows. The existing `S` spawn and `F` flag positions move proportionally. Pick a "ramp + descent + finale" layout so scrolling reveals new content.
7. **Parallax backgrounds with 3 layers** in screen-space using `fixed()` and per-frame manual offset, not Kaplay's built-in (none for parallax in v3001). Layers: gradient sky, clouds (drift slowly + parallax), hills silhouette (parallax only).
8. **Camera dead-zone follow**: ±80 px from camera-center. Camila moves freely inside the band; camera catches up only when she crosses it. Plus the existing left/right level-edge clamp.
9. **HUD with three zones**: top-left = nugget count (small nugget icon + Press-Start-2P number), top-right = lives (small Camila-head icon + ×N), top-center = level name "WORLD 1-1: KITCHEN ROAD" that fades after ~2 s.
10. **No new game-logic features** — no scoring system, no nugget counter persistence between deaths, no level select, no extra enemies. The HUD's "nugget count" is a per-run counter that resets on scene reload, same as lives. Anything else is post-v0.2.

### Phase B — Pixel-art helper + hand-authored sprites (~45 min)

**B1. `src/pixel-art.js`** — exposes:
```js
// rows: ["..#.", "####", ".#.#"] etc — each char is a palette key.
// palette: { '.': null, '#': '#1a1a1a', 'A': '#ffffff', ... } (null = transparent).
// Returns an HTMLCanvasElement Kaplay can register via loadSprite.
export function makePixelCanvas(rows, palette) { ... }

// Convenience: register a multi-frame animation from N row-arrays.
// Returns a single horizontal-strip canvas; loadSprite() with sliceX = N reads frames.
export function makePixelStrip(framesArrays, palette) { ... }
```

  Implementation: `document.createElement('canvas')`, set `width`/`height`, get `2d` context, set `imageSmoothingEnabled = false`, `fillRect(x, y, 1, 1)` per pixel. ~25 lines.

**B2. Hand-authored sprite definitions.** Each file in `src/art/` exports `{ frames, palette, anims }`.   Concrete grids:

    - `camila-tall-walk` — alt-frame variant.

### Phase D — Parallax background (~30 min)

**`src/parallax.js`:**
```js
// Spawns three z-stacked sprite layers inside the current scene.
// Layers are placed with `fixed()` so they live in screen-space; we manually
// offset their positions each frame based on camera X * a layer-specific factor.
export function spawnParallax({
  skyTiles,    // array of frame indices for backgrounds.png — drawn as horizontal strip
  cloudFrames, // array of (frame, y, baseX, drift) — independent floaters
  hillFrames,  // array of frame indices, tiled across the bottom
})
```

Layers:
- **Sky (z = -100)**: a vertical gradient using 1–2 background tiles at full-canvas-width (Kenney bg tiles repeat horizontally; we tile them along the full canvas width). `fixed()` so it never moves. No parallax — it's the static back wall.
- **Clouds (z = -50)**: 3–4 cloud sprites placed at semi-random heights across screen-width. Each cloud's `pos.x` updates as `baseX - camera.x * 0.25` (so when the camera moves right by 100 px, clouds move left by 25 px → camera-relative drift of 75 px). Plus a tiny sin-wave bob `+= sin(time() * 0.5) * 0.5`.
- **Hills (z = -25)**: a row of green hill silhouette tiles tiling the full screen width. Parallax factor `0.5`. Sits just above the visible ground row.

All three layers respect Kaplay's `fixed()` so the `camPos()`-driven world transform doesn't auto-translate them; we apply parallax math ourselves.

### Phase E — Camera dead-zone (~15 min)

**`src/camera.js`:**
```js
export function makeDeadZoneCamera({ target, levelWidth, deadZone = 80, lerp = 0.15 }) {
  let camX = target.pos.x
  return {
    update() {
      const off = target.pos.x - camX
      if (off > deadZone)        camX = target.pos.x - deadZone
      else if (off < -deadZone)  camX = target.pos.x + deadZone
      const halfW = width() / 2
      const clamped = Math.min(Math.max(camX, halfW), Math.max(halfW, levelWidth - halfW))
      // Optional smoothing — comment out if it feels rubbery on iPad.
      const cur = camPos()
      camPos(cur.x + (clamped - cur.x) * lerp, height() / 2)
    },
  }
}
```

Replace the existing camera-follow `onUpdate` in `level1.js` with `cam.update()`. Tunable `deadZone` exposed in `config.js` so Camila can change it.

## Risks / things to watch

1. **Hand-authored sprite quality**: a 16×24 chef coat drawn by string-pixel-arrays will look stiff. I'll only know it's right by screenshotting and showing the user. Plan to iterate at least once on Camila and the notebook after first render.
2. **Tile-frame numbers** in Kenney's pack are not documented — implementor must visually pick from `Preview.png`. There's a 5-minute "open in Finder, count to slot N" task hidden in Phase A2.
3. **Animation system in Kaplay v3001** — `loadSprite(name, src, { sliceX, sliceY, anims })` and `obj.play("anim_name")` are the canonical APIs but I haven't re-tested them in v3001 specifically. Fallback: manual frame-index swap each tick if `play()` doesn't loop reliably.
4. **iPad pixel-perfect**: `image-rendering: pixelated` works in Safari ≥ 14. Older iPads might show blurry tiles even with the CSS — known Safari quirk, no workaround except `crisp-edges`. Test on the actual demo iPad.
5. **Bundle size** — Press Start 2P TTF is ~30 KB, Kenney sheets are ~50 KB total, hand-authored sprites are negligible. We should still come in under v0.1's ~280 KB images budget by a comfortable margin.
6. **Camila's body still uses a `rect()` for collision** (auto-area). When we switch to `sprite()`, the area derives from the sprite's pixel bounds, which can include transparent margins. We may need to pass a custom `area({ shape: ... })` again — but this time with `anchor('topleft')` (default) to avoid the v0.1 floor-sinking bug that was caused by `anchor('bot')` interacting weirdly with custom shapes. Lesson learned.
7. **"Don't refactor game logic"** is a real constraint — be disciplined. Maker functions can change their RENDER, but their TAGS, AREA, and BODY components stay structurally identical so collision handlers don't move.
8. **Level length matters for camera dead-zone**: if the level isn't actually 3-4 screens wide the dead-zone effect is invisible. Phase H is required, not optional.
